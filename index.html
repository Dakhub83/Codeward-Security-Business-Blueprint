Security Best Practices

What good security looks like, across every layer

Practical, engineering-first guidance for businesses and technical teams — no vendor lock-in, no scare tactics.

01

Never store secrets in code

API keys, database passwords, and certificates hardcoded in source files are the single most common cause of credential leaks. Use environment variables and a secrets manager — every cloud provider offers one.

AppSec & Code Audit

02

Scan dependencies before merging

Your application is as vulnerable as its weakest library. Automated Software Composition Analysis (SCA) flags known CVEs in open-source packages before they reach production, not after a breach.

AppSec & Code Audit

03

Validate input at every boundary

Never trust data from users, APIs, or external services. Validate length, type, format, and range at every entry point. Injection flaws — SQL, command, LDAP — are always the result of trusting unvalidated input.

AppSec & Code Audit

04

Use parameterized queries, always

String-concatenated SQL is one of the oldest known vulnerabilities and still among the most exploited. Parameterized queries and prepared statements eliminate the entire class of SQL injection — there is no acceptable excuse for not using them.

AppSec & Code Audit

05

Security-review authentication logic manually

Automated scanners miss business-logic flaws in login, session management, and password-reset flows. A human engineer familiar with auth attack patterns — brute force, token replay, account enumeration — must review these flows before launch.

AppSec & Code Audit

06

Keep a Software Bill of Materials

An SBOM is an inventory of every library and component your software contains. When a zero-day like Log4Shell surfaces, organizations with an SBOM know within minutes whether they're exposed. Those without one spend weeks guessing.

DevSecOps

01

Run security checks on every commit

Security testing at the end of a release cycle finds flaws when they're expensive to fix. A SAST gate on every pull request finds them when a developer is still in context and the fix takes minutes, not days.

DevSecOps

02

Fail builds on critical findings

A scanner that generates a report nobody reads is theater. Blocking deployments automatically on verified critical or high-severity findings ensures security has teeth — configure it with a path to exception handling so legitimate urgency doesn't bypass the gate.

DevSecOps

03

Scan infrastructure-as-code before apply

Terraform, CloudFormation, and Pulumi misconfigurations are one of the top cloud breach vectors. Tools like Checkov and tfsec catch open S3 buckets, missing encryption, and excessive IAM permissions before they're ever provisioned.

DevSecOps

04

Sign and verify release artifacts

Supply-chain attacks insert malicious code between build and deploy. Signing artifacts with Sigstore/Cosign and verifying signatures at deployment time ensures that what you built is exactly what you shipped — nothing more was added in transit.

DevSecOps

05

Rotate credentials on a schedule

Credentials that never rotate are credentials that inevitably leak. Automate rotation for database passwords, API keys, and service accounts — and revoke the previous value immediately rather than leaving it valid "just in case."

DevSecOps

06

Track MTTR alongside velocity

Mean-time-to-remediate is a security KPI as meaningful as deployment frequency. Teams that measure it hold themselves accountable for closing findings, not just discovering them — and tend to fix vulnerabilities 3–5× faster than teams that don't.

Continuous Monitoring

01

Segment your network by risk level

Payment systems, production servers, and employee laptops should never share a flat network. Segmentation with VLANs or software-defined perimeters limits blast radius — a compromised workstation can't reach your database if they're on separate segments.

Network & Cloud Security

02

Replace VPN-only access with zero-trust

Traditional VPNs grant broad network access once authenticated. Zero-trust network access (ZTNA) grants access only to specific applications based on user identity, device health, and context — lateral movement becomes structurally much harder.

Network & Cloud Security

03

Monitor east-west traffic, not just the perimeter

Most breaches involve lateral movement inside the network after an initial foothold. Monitoring traffic between internal systems — not just traffic entering and leaving — detects attackers who have already bypassed the perimeter.

Continuous Monitoring

04

Audit firewall rules quarterly

Firewall rules accumulate over time. Rules added for a temporary project rarely get removed. A quarterly audit removes stale rules, tightens overly-broad permissions, and reduces attack surface that nobody is actively defending.

Network & Cloud Security

05

Enforce MFA on all remote access, no exceptions

Stolen passwords are the most common initial access vector. Multi-factor authentication is the single highest-ROI security control for remote access — it blocks the vast majority of credential-based attacks even when passwords are fully compromised.

Network & Cloud Security

06

Test your API endpoints for authorization flaws

Broken Object Level Authorization (BOLA) lets one authenticated user access another user's data by simply changing an ID in a request. It's the most common API vulnerability and rarely detected by automated scanners — it requires manual testing by someone thinking like an attacker.

Network & Cloud Security

01

Apply least-privilege IAM from day one

No user, service account, or Lambda function should have more permissions than it needs. Start with minimum permissions and expand when needed — overly-broad IAM roles are among the top cloud misconfigurations exploited in breaches.

Network & Cloud Security

02

Enable audit logging in every region

CloudTrail, Azure Monitor, and GCP Audit Logs are cheap to enable and invaluable during incident response. Enable them in every region — not just the ones you think you're using — because attackers intentionally spin up resources in quiet regions to avoid detection.

Continuous Monitoring

03

Scan IaC before provisioning

Infrastructure misconfigurations are consistently the #1 cloud breach vector. Running Checkov or tfsec in CI before terraform apply catches public S3 buckets, unencrypted databases, and open security groups before they're ever created.

DevSecOps

04

Encrypt data at rest and in transit

Use your cloud provider's KMS for encryption at rest on every storage service. Enforce TLS 1.2+ for all data in transit — and revoke support for older protocols explicitly rather than hoping they're disabled by default.

Network & Cloud Security

05

Audit public-facing storage monthly

Publicly accessible S3 buckets and Azure Blob containers have exposed sensitive data at virtually every major cloud provider. Run a monthly audit of public access settings — and turn on AWS S3 Block Public Access or its equivalent at the account level as a guardrail.

Network & Cloud Security

06

Harden container images before deployment

Use minimal base images, run containers as non-root, scan images for CVEs before pushing to a registry, and enforce signed images at deployment. An unpatched container image is as dangerous as an unpatched server — with the added risk of running at scale.

DevSecOps

01

Map controls to frameworks early

Retrofitting compliance onto a system built without it is 3–5× more expensive than building it in from the start. Whether you're targeting SOC 2, PCI-DSS, ISO 27001, or DORA — understand the control requirements before you architect, not during the audit.

GRC & Compliance

02

Automate evidence collection

Manual evidence gathering — screenshots, exported logs, email trails — does not scale beyond your first audit. Platforms like Vanta and Drata continuously collect evidence and map it to controls automatically. The cost of automation is a fraction of the cost of a failed audit.

GRC & Compliance

03

Know your breach notification deadlines

GDPR, DORA, Nigeria's NDPA, and Kenya's Data Protection Act all mandate breach notification within 72 hours of becoming aware of an incident. Missing this deadline is a separate regulatory violation from the breach itself — your incident response plan must include notification steps and pre-drafted templates.

GRC & Compliance

04

Conduct quarterly risk assessments

Annual risk assessments miss the threats that emerged in months two through eleven. A lightweight quarterly review — what changed in our environment, our vendor relationships, and the threat landscape — keeps your risk register actionable rather than a document that ages in a folder.

GRC & Compliance

05

Vet your vendors the same way you vet your code

Under DORA and NIS2, regulated entities are explicitly accountable for the cyber posture of their ICT third-party providers. A vendor questionnaire submitted once at onboarding is not third-party risk management — it requires ongoing monitoring, contractual obligations, and regular reassessment.

GRC & Compliance

06

Test your incident response plan before you need it

A plan that has never been tested is a hypothesis, not a capability. Run tabletop exercises at least twice a year — simulate a ransomware attack, a credential breach, a third-party outage — and update the plan based on what the exercise reveals.

Continuous Monitoring

01

Change default credentials at installation

IP cameras, access controllers, and alarm panels ship with default usernames and passwords that are publicly documented. Leaving them in place is the physical-security equivalent of leaving your front door unlocked — and these devices are internet-reachable.

Physical Security

02

Patch physical security firmware

Cameras and access controllers run embedded operating systems with real CVEs that go unpatched for months or years. Treat physical security devices the same way you treat servers — subscribe to vendor security bulletins and patch on a defined schedule, not only when something breaks.

Physical Security

03

Isolate physical security from corporate IT

Your IP camera network should not share a subnet with your corporate laptops. A compromised camera on a flat network can be used as a pivot point into your file servers. Place physical security devices on a dedicated, firewalled VLAN with strict ingress and egress rules.

Physical Security

04

Log and review physical access

Access control systems generate detailed logs of who entered which door, when. Review server-room and sensitive-area access logs monthly — unusual after-hours access or repeated failed attempts are early indicators of insider threat or reconnaissance activity.

Physical Security

05

Use multi-factor access control for sensitive areas

A single access card can be cloned or stolen. Server rooms, data centers, and vaults should require a second factor — PIN, biometric, or time-of-day restriction — so that possessing a card alone is not sufficient for entry.

Physical Security

06

Correlate physical and digital alerts

An after-hours door access followed immediately by an unusual login from the same user is more meaningful than either alert alone. Integrating your physical access logs with your SIEM enables this correlation and catches attack patterns that siloed monitoring misses entirely.

Continuous Monitoring

01

Train before the breach, not after

Security awareness training delivered reactively — after a phishing attack or a compliance finding — reaches people who are already demoralized. Build training into onboarding and run regular refreshers, so good habits form before they're tested by an actual attack.

Security Training

02

Run realistic phishing simulations

Real phishing emails are tailored, timely, and convincing. Generic simulations that every employee recognizes do not measure real risk. Use sector-specific scenarios — fake vendor invoices, fake HR policy updates, fake regulatory notices — and measure both click rates and reporting rates.

Security Training

03

Build a no-blame reporting culture

Employees who fear punishment for clicking a phishing link don't report it — and a breach that isn't reported isn't contained. Create a clear, low-friction channel for reporting suspicious activity and actively reinforce that reporting is the right behavior, not an admission of failure.

Security Training

04

Brief executives separately

Executives are the highest-value targets for business email compromise, spear-phishing, and deep-fake voice fraud. They also tend to skip generic training. Run dedicated briefings that walk through real attack scenarios targeting their role, authority level, and communication patterns.

Security Training

05

Teach developers secure coding, not just awareness

Generic phishing training is not what developers need. Engineers benefit from hands-on secure coding bootcamps — OWASP Top 10, secrets management, dependency hygiene — delivered in the context of their actual stack, not hypothetical scenarios disconnected from their daily work.

Security Training

06

Document training for your auditors

SOC 2, PCI-DSS, HIPAA, and ISO 27001 all require documented security awareness training for employees. Maintain completion records, training materials, and assessment results — not just a calendar invite with a "done" checkbox. Auditors ask for evidence, not assurances.
